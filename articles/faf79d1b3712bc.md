## はじめに

`418 I'm a teapot` というHTTPステータスコードをご存知でしょうか。[RFC 2324](https://developer.mozilla.org/ja/docs/Web/HTTP/Reference/Status/418)（Hyper Text Coffee Pot Control Protocol）という、1998年のエイプリルフールに公開されたジョークRFCで定義されたステータスコードです。「サーバーはティーポットであるため、コーヒーを淹れることを拒否する」という、なんとも遊び心のある仕様です。

今回、この418ネタをただのデコレーションではなく**実際に動くHTTPレスポンスとして返す**静的ページを作り、Cloudflare Pagesにデプロイしました。本記事ではそのセットアップ手順と、途中でハマった点を共有します。

## 目的

- Cloudflare Pages Functionsを使ってみたかった
- ネタページをミニマムに作りたかった。
- GopherとFerrisが可愛い

$ curl -I https://teapot.taka1156.site/

## 構成

```txt
.
├── index.html              # ティーポットページ本体
├── public
│   ├─── img
│   │   ├──  i_am_a_teapot.webp # gopherとferrisがお茶してる画像
│   │   └──  各種アイコン      　　# GitHubやZenn画像
│   └─ favicon.png
├── functions/
│   └── index.js            # ステータスを418に書き換えるPages Functions
├── wrangler.toml
└── package.json
```

ページ自体はシンプルな静的HTML/CSSです。見どころはステータスコードを実際に418へ書き換える部分にあります。

## なぜステータスコードの書き換えが必要か

CloudflarePages（やNetlify、Vercelなどの静的ホスティング全般）は基本的に200番台のレスポンスを返す前提で作られています。ダッシュボードの設定だけでは、特定のページのレスポンスコードを任意の値に変えることはできません。

そこで使うのが **Cloudflare Pages Functions** です。これはPagesプロジェクトに同梱できる軽量なサーバーサイド処理（Cloudflare Workersのサブセット）で、リクエストをインターセプトしてレスポンスを加工できます。

## セットアップ手順

### 1. Pages Functionsを書く

`functions/index.js` にルート（`/`）へのリクエストをハンドリングする関数を置きます。

```javascript:functions/index.js
export async function onRequest(context) {
  const response = await context.env.ASSETS.fetch(context.request);

  return new Response(response.body, {
    status: 418,
    statusText: "I'm a teapot",
    headers: response.headers,
  });
}
```

`context.env.ASSETS.fetch()` で静的アセット（`index.html`など）の中身をそのまま取得し、ステータスコードだけ418に差し替えて返しています。
`functions`フォルダの中にファイルを置くだけで、ファイルパスに対応したルーティングが自動的に行われるのがPages Functionsの特徴です。
（`functions/index.js`なら`/`、`functions/teapot.js`なら`/teapot`に対応）

### 2. wrangler.tomlを用意する

ローカル開発（`wrangler pages dev`）や設定の明示化のために、プロジェクトルートに置きます。

```toml:wrangler.toml
name = "teapot"
compatibility_date = "2026-06-19"
pages_build_output_dir = "."
```

`compatibility_date`を指定しないと、実行のたびにWranglerが警告を出してくるので明示しておくのが無難です。

### 3. package.jsonにWranglerを入れる

ローカルでの動作確認用に、`devDependencies`としてWranglerを追加します。

```json:package.json
{
  "name": "teapot",
  "version": "1.0.0",
  "scripts": {
    "dev": "wrangler pages dev --port 4418 --ip 0.0.0.0 ."
  },
  "devDependencies": {
    "wrangler": "^4.103.0"
  }
}
```

`--ip 0.0.0.0` を付けているのは、後述するdevcontainer環境でホストOSのブラウザからアクセスできるようにするためです。

### 4. ローカルで動作確認

```bash
yarn install
yarn dev
```

```bash
curl -I http://localhost:4418/
```

`HTTP/1.1 418` が返ってくれば成功です。

### 5. Cloudflareダッシュボードでデプロイ

GitHubリポジトリをpushした上で、Cloudflareダッシュボードから設定します。

1. **Workers & Pages** → **Create application**
2. 必ず **Pages** タブを選んでから **Connect to Git**
3. リポジトリを選択
4. ビルド設定は以下の通り（ビルドステップ不要な静的サイトのため）

| 項目 | 値 |
| --- | --- |
| Build command | （空欄） |
| Build output directory | `/` |

`functions/`フォルダはルート直下に置いてあれば、Git連携の場合は自動で検出・デプロイされます。

デプロイ後、本番URLで確認します。

```bash
curl -I https://your-project.pages.dev/
```

`HTTP/2 418` が返れば完了です。

## つまづいたポイント

ここからは実際にハマった箇所を簡単に紹介します。

### Alpine LinuxではWranglerのworkerdが動かない

開発をdevcontainer（Alpine Linuxベース）で行っていたところ、`yarn dev`実行時に以下のエラーが出ました。

```shell
Error: spawn /workspace/node_modules/@cloudflare/workerd-linux-arm64/bin/workerd ENOENT
```

ファイルは実在し、実行権限も付いているにも関わらず`ENOENT`になる場合、**バイナリが要求する動的リンカが存在しない**ケースを疑う必要があります。

:::message
Wranglerが内部で使用する`workerd`（Cloudflare WorkersのランタイムであるV8ベースのネイティブバイナリ）はglibcを前提にビルドされています。muslベースのAlpine Linuxではこの動的リンカが無いため、ファイルが存在していても実行できません。
:::

対処法としては以下のいずれかになります。

- devcontainerのベースイメージをDebian/Ubuntu系（`-slim`など）に変更する（確実）
- `apk add gcompat` でglibc互換レイヤーを足す（手軽だが、複雑なバイナリでは不十分なこともある）

今回はベースイメージをslim系に変更して解決しました。

---

### Cloudflareダッシュボードでの作成フローでWorkers扱いになることがある

ローカルでの動作確認後、Cloudflareダッシュボードの「Connect to Git」からリポジトリ連携してデプロイしたところ、ビルドログに以下のエラーが出ました。

```shell
✘ [ERROR] Missing entry-point to Worker script or to assets directory
```

このエラーは本来`wrangler pages deploy`が実行されるべきところで、**`wrangler deploy`（無印、Workers用）が実行されてしまっている**ことを示しています。

よく読んで、下をクリックしましょう。(3敗)
自分は、GitHubの連携ボタンを見つけた瞬間にそこを押してました。
![cloudflareダッシュボード](/images/cloudflare-dash.png)

このケースでは一度プロジェクトを削除し、改めてPagesから作成し直すことで解決しました。
