## 対象読者

- Cloudflare Tunnel を使って VPS 上のサービスを公開しようとしている方
- 既存のトンネルに新しいサービスのルールを追加したのに 404 が返ってくる方
- `config.yml` の ingress ルールを編集したのに反映されない方
- `cloudflared tunnel route dns` で DNS を登録済みなのに疎通しない方

説明上、ドメイン名は念の為、仮の記載にしています。

## 背景

ConoHa VPS の管理には [bastion-ssh](https://github.com/taka1156/bastion-ssh) を使っており、Dev Container + Cloudflare Tunnel 経由で SSH 接続しています。
今回は稼働中の同じトンネル（`my-tunnel`）に新しい Web サービスのルールを追加しようとしたところ、`404` エラーが返り続けました。その調査・解決の記録です。

同じトンネルに別サービスを載せようとして 404 で詰まった方に参考になれば。

## そもそもなんでVPSでトンネルを経由するの？

ConoHaというかVPS全般で、DDosを受けてしまうと、通信遮断されて対策するまで復旧してもらえない問題があります。

対策としては、以下がありますが、設定ファイルが複数の場所に散らばりやすく、時間が経つと設定箇所を忘れる可能性がありました。

- fail2banを利用する
- GeoIPで国内アクセスを弾く
- nginxのレートリミットをかける

その辺りの簡略化を図るために、Cloudflareに丸投げして解決しました。

また、仕組み上、インバウンドトラフィックのポートを全部封じることができるのも不正アクセスを防ぐという意味で良い選択だと考えています。

## 環境

| 項目 | 内容 |
| ------ | ------ |
| VPS | ConoHa VPS（GMOインターネットグループ株式会社） |
| cloudflared | Cloudflare Tunnel エージェント |
| SSH 接続 | [bastion-ssh](https://github.com/taka1156/bastion-ssh)（Dev Container ベース） |
| ドメイン | `example.com`（Cloudflare 管理） |

## 目標

独自ドメインのサブドメインから、VPS 上で動いている Web サービスにアクセスします。

bastion-ssh で利用している SSH 用のトンネルはすでに稼働中であり、同じ `my-tunnel` に新しいルールを追加するだけで済むはずだと考えていました。

## 発生した問題

### 症状

```shll
HTTP/2 404
server: cloudflare
```

DNS は Cloudflare に届いているが、404 が返ります。

### 調査

```shell
# 追加したいサービスの稼働確認
# → 対象サービスは正常稼働中 ✅

# cloudflared の稼働確認
systemctl status cloudflared
# → active (running) ✅

# DNS CNAME の確認
cloudflared tunnel route dns <tunnel-id> service.example.com
# → is already configured to route to your tunnel ✅
```

すべて正常に見えたが、cloudflared のログを確認すると根本原因が判明しました。

```shell
sudo journalctl -u cloudflared -n 30 --no-pager
```

```shll
INF Updated to new configuration config={"ingress":[
  {"hostname":"ssh.example.com","service":"ssh://localhost:22"},
  {"service":"http_status:404"}
]}
```

ログを見ると、追加したはずの `service.example.com` のルールが存在せず、SSH のルールしか登録されていないことが分かりました。

## 根本原因

### ローカル設定とリモート設定の競合

cloudflared には設定の管理方法が2つあります。

| 方式 | 説明 |
| ------ | ------ |
| **ローカル設定** | `/etc/cloudflared/config.yml` に ingress ルールを記述 |
| **リモート設定** | Cloudflare Zero Trust ダッシュボードで管理（API 経由で自動取得） |

このトンネルは **リモート設定が有効**になっており、ローカルの `config.yml` に書いた ingress ルールは**完全に無視**されていました。

リモート設定には bastion-ssh のセットアップ時に登録した `ssh.example.com`（SSH）しか存在していませんでした。
また、新たに追加したルールがなかったため、フォールバックの `http_status:404` が返ってます。

### 自分のトンネルがどちらのモードか確認する方法

`journalctl` のログで判断できます。

**リモート設定が有効な場合のログ（今回のケース）:**

```shell
INF Updated to new configuration config={...}
```

ダッシュボードから取得した設定で動作中であることを示すようです。

**ローカル設定が有効な場合のログ:**

```shell
INF Loading configuration from file config=/etc/cloudflared/config.yml
```

または、`config.yml` の中身でも判断できます。

- `ingress:` ブロックがある → ローカル設定モード
- `tunnel:` と `credentials-file:` のみで `ingress:` がない → リモート設定モード

> **注意**: `config.yml` 自体は削除してはいけない。`tunnel:`（トンネル名またはID）と `credentials-file:`（認証情報ファイルのパス）は cloudflared の起動・認証に必要であり、ingress ルールをリモートで管理していても参照される。

## 解決手順

### 1. Cloudflare Zero Trust で Published application routes を追加

ローカルの `config.yml` を編集しても意味がないようです。**Zero Trust ダッシュボードからルールを追加しました**。

[Cloudflare Zero Trust](https://one.dash.cloudflare.com/)

```shell
→ **Networks**
→ **Tunnels**
→ `my-tunnel`
→ **Edit**
→ **Published application routes**
→ **Add a published application routes**
```

| 項目 | 値 |
| ------ | ----- |
| Subdomain | 任意のサブドメイン |
| Domain | `example.com` |
| Type | サービスに合わせて選択（`HTTP` / `HTTPS` など） |
| URL | `localhost:<ポート番号>`<br>今回Dockerへのアクセスなので |

設定保存後、cloudflared に自動で反映される（再起動不要）。

### 2. DNS レコードの重複エラーへの対処

**`cloudflared tunnel route dns` コマンドで事前に CNAME を作成していた場合**

ダッシュボードから Published application routes を追加しようとするとエラーになりました。

```shell
Error: An A, AAAA, or CNAME record with that host already exists.
```

**対処法:**

1. [Cloudflare ダッシュボード](https://dash.cloudflare.com/) → 対象ドメイン → **DNS** → **Records** → 該当サブドメインの CNAME を**削除**
2. 再度 Zero Trust から Published application routes を追加（DNS レコードは自動作成される）

## 結果

設定保存後、cloudflared に自動で反映され（再起動不要）、目的のサービスにアクセスできるようになりました。

反映されたことは `journalctl` で確認できます。

```shell
sudo journalctl -u cloudflared -n 30 --no-pager
```

```shell
INF Updated to new configuration config={"ingress":[
  {"hostname":"ssh.example.com","service":"ssh://localhost:22"},
  {"hostname":"service.example.com","service":"http://localhost:8080"},
  {"service":"http_status:404"}
]}
```

追加したサブドメインのルールが ingress に含まれていれば反映済み。

## まとめ・教訓

- **リモート設定が有効なトンネルでは、`config.yml` の ingress ルールは無視される**
  - ルールの追加・変更は必ず Zero Trust ダッシュボードで行う
- **`cloudflared tunnel route dns` で手動登録した DNS は、ダッシュボードの Published application routes 追加時に競合する**
  - DNS レコードを先に削除してから追加し直す
- bastion-ssh で SSH トンネルを使っている場合、同じトンネルに別サービスのエントリを追加できる。ただし追加はダッシュボード側で行うこと

## おまけ: Portainer（HTTPS / 自己署名証明書）を追加する場合

Portainer CE はデフォルトでポート `9443` に HTTPS（自己署名証明書）で稼働している。この場合、Published application routes の追加時に **No TLS Verify** を有効にする必要があります。

| 項目 | 値 |
| ------ | ----- |
| Type | `HTTPS` |
| URL | `localhost:9443` |
| No TLS Verify | **オン** |

> cloudflared はデフォルトでバックエンドの TLS 証明書を検証する。自己署名証明書のままでは接続を拒否されるため、No TLS Verify が必須。
