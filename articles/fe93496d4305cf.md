## はじめに

リモートサーバーへの SSH 接続には、秘密鍵や `~/.ssh/config` の管理が欠かせません。

しかし、複数のマシンや複数のプロジェクトで使い回すと、ホストマシンに認証情報が散在してしまいます。

特に、ホストPCの隠蔽されているディレクトリは魔境になりがちなのでそれは避けたいと考えていました。

**bastion-ssh** はその課題を Dev Container で解決します。SSH の認証情報と接続ロジックをすべてコンテナ内に閉じ込め、ホストマシンを一切汚さずにリモートサーバーへ接続できます。

## bastion-ssh とは

[bastion-ssh](https://github.com/taka1156/bastion-ssh) は、Dev Container をベースにした汎用 SSH 踏み台ツールです。

- **ホスト環境ゼロ汚染** — SSH 鍵、設定ファイルの両方ともホストには書かれない
- **どこでも同じワークフロー** — Dev Containers が動く環境ならどこでも同一手順で動作
- **GitHub Codespaces 対応** — ブラウザだけで完結させることも可能
- **Cloudflare Tunnel 対応** — VPS の SSH ポートをインターネットに直接公開せずに接続可能

## 仕組み

```shell
┌────────────────────────────────┐
│  Dev Container (bastion-ssh)   │
│                                │
│  tool/.env  ←  make setup      │
│  SSH key                       │
│                                │
│  make ssh  ──────────────────────────→  Remote Server
│  make rsync                    │        (直接 or Cloudflare Tunnel 経由)
│  make sftp                     │
└────────────────────────────────┘
```

すべての操作はコンテナ内で完結します。ホスト側に SSH エージェントや鍵ファイルを用意する必要はありません。

## セットアップ

### 1. sshキーをプロジェクトの key ディレクトリ(.key)に配置する

### 1. リポジトリを Dev Container で開く

VS Code に [Dev Containers 拡張機能](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-containers) をインストールし、リポジトリを Dev Container として開きます。

### 2. 接続情報を設定する

対話形式のセットアップコマンドを実行すると、`tool/.env` が生成されます。
※ ARGSでファイル名を指定できます

```shell
make setup ARGS=.env.test

=== SSH Settings ===
HOST (IP address) [xxx.xxx.xxx.xxx]: 255.255.255.255
USER [hoge]: sample
PORT [22]: 2222
KEY (path to private key) [/root/.ssh/your-key.pem]: sample/your-key.pem

=== Cloudflare Tunnel (optional) ===
Enable Cloudflare Tunnel? [y/N]: y #　利用しない場合、skip可能です
SUBDOMAIN: sub
DOMAIN: domain

Saved to .env.test
```

プロンプトに従って以下の情報を入力します。

| 変数 | 説明 |
| ------ | ------ |
| `HOST` | 接続先サーバーの IP アドレス |
| `USER` | SSH ログインユーザー名 |
| `PORT` | SSH ポート番号（デフォルト: 22） |
| `KEY` | コンテナ内の秘密鍵パス |

> 秘密鍵のパーミッションエラーが出た場合:
>
> ```shell
> chmod 600 /root/.ssh/your-key.pem
> ```

### 3. 接続する

```shell
make ssh
```

これだけです。`ssh` コマンドのオプションを覚える必要も、`~/.ssh/config` を編集する必要もありません。

## ファイル転送

SSH 接続と同じ `.env` 設定を使って、rsync や sftp でのファイル転送も行えます。

### rsync

```shell
# ローカル → リモート（アップロード）
make rsync ARGS="-av ./local/path/ user@host:/remote/path/"

# リモート → ローカル（ダウンロード）
make rsync ARGS="-av user@host:/remote/path/ ./local/path/"
```

### sftp

```shell
make sftp ARGS="user@host"
```

## オプション: Cloudflare Tunnel 経由で接続する

VPS の SSH ポートをインターネットに直接公開したくない場合、Cloudflare Tunnel を利用できます。`make setup` 時にトンネル有効化を選択するだけで、`tool/.env` に以下が追加されます。

```shell
ENABLE_TUNNEL=true
SUBDOMAIN=subdomain
DOMAIN=example.com
```

設定後は `make ssh` / `make rsync` / `make sftp` のすべてのコマンドが自動的に Cloudflare Tunnel 経由で動作します。追加の手順は不要です。

**接続の流れ:**

```shell
# トンネルなし
ssh -i $KEY -p $PORT $USER@$HOST

# トンネルあり（ProxyCommand で自動ラップ）
ssh -i $KEY -p $PORT \
  -o ProxyCommand="cloudflared access ssh --hostname $SUBDOMAIN.$DOMAIN" \
  $USER@$HOST
```

## まとめ

bastion-ssh を使うことで以下が実現できます。

- ホストマシンに SSH 認証情報を残さない
- チームメンバー間でまったく同じ接続手順を共有できる
- `make ssh` の 1 コマンドで接続できる簡潔なワークフロー
- Cloudflare Tunnel によるセキュアな接続をオプションで追加可能

Dev Container や Codespaces を普段使っている方は、ぜひ試してみてください。

## 今後の検討事項

現在はシェルスクリプトと `.env` ファイルによるシンプルな構成ですが、以下の拡張を検討しています。

:::details  `~/.ssh/config` によるマルチホスト管理

### `~/.ssh/config` によるマルチホスト管理

現状は `.env` の `KEY` 変数に単一の秘密鍵パスを指定する方式のため、接続先サーバーや用途ごとに鍵を使い分けたい場合はその都度 `.env` を編集する必要があります。

SSH 標準の `~/.ssh/config` をコンテナ内にマウントする方式への移行を検討しています。`~/.ssh/config` はホスト側ではなくコンテナ内にのみ存在するため、bastion-ssh の根本思想（ホストを汚さない）とは矛盾しません。

```shell
# コンテナ内の ~/.ssh/config（ホストには存在しない）
Host production
  HostName xxx.xxx.xxx.xxx
  User ec2-user
  IdentityFile /root/.ssh/prod.pem

Host staging
  HostName yyy.yyy.yyy.yyy
  User ubuntu
  IdentityFile /root/.ssh/staging.pem
  ProxyCommand cloudflared access ssh --hostname staging.example.com
```

ファイル管理は現在の `.env` / `.env.example` と同じパターンで対応できます。

```shell
tool/
  ssh_config.example   ← リポジトリで管理（テンプレート）
  ssh_config           ← .gitignore に追加（実際の設定）
```

なお、`~/.ssh/config` 方式に移行する場合も、`make setup` による対話形式の初期設定は引き続き提供したいと考えています。
手動でファイルを編集しなくてもセットアップが完了できる体験は維持する方針です。

:::

:::details シェルスクリプトの Go バイナリ化

### シェルスクリプトの Go バイナリ化

現在の `tool/*.sh` はシェルスクリプトで実装されています。Go で書き直してシングルバイナリとしてビルドすることで、以下のメリットが期待できます。

- Dev Container に依存せず、バイナリ単体で動作可能
- クロスプラットフォームなビルドが容易（macOS / Linux / Windows）
- テストやエラーハンドリングの記述がシェルより簡潔になる

:::
